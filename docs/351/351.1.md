# Why Virtualization?

Modern servers are rarely dedicated to a single operating system.

Virtualization allows multiple isolated operating systems to share the same physical hardware while remaining independent from each other.

Typical benefits include:

- better hardware utilization
- workload consolidation
- isolation between workloads
- snapshots
- live migration
- easier backup and disaster recovery
- test and development environments

Typical disadvantages or challenges include:

- additional complexity
- performance overhead
- shared physical resources
- more complex troubleshooting
- dependency on the virtualization host
- licensing and support considerations

> **Exam Tip**
>
> LPIC may ask about both benefits and drawbacks of virtualization.
> Do not only memorize the advantages.

# What is a Virtual Machine?

A **Virtual Machine (VM)** is a software-defined computer that behaves like a physical computer.

From the perspective of the operating system running inside the VM, the **guest operating system**, the virtual machine appears to provide its own hardware, including:

- one or more virtual CPUs (vCPUs)
- virtual memory (RAM)
- virtual storage devices
- virtual network interfaces
- virtual firmware such as BIOS or UEFI

Although these resources appear to belong exclusively to the virtual machine, they are provided by virtualization software and ultimately backed by the physical hardware of the host system.

This allows multiple isolated operating systems to run simultaneously on a single physical computer.

---

## Host and Guest

Virtualization introduces two important terms:

- **Host**: The physical system that provides CPU, memory, storage and other hardware resources.
- **Guest**: The operating system running inside a virtual machine.

One host system can run multiple guest operating systems at the same time.

---

## Virtual Hardware

A guest operating system interacts with virtual hardware in the same way it would interact with physical hardware.

Examples include:

- virtual CPU
- virtual RAM
- virtual disk
- virtual network adapter
- virtual graphics adapter
- virtual firmware

The virtualization layer translates or maps these requests to the underlying physical hardware.

> **Common Mistake**
>
> A virtual machine is **not** simply a collection of files.
>
> The virtual disk (`.qcow2`, `.vmdk`, etc.) is only one part of a VM. CPU state, memory allocation, virtual devices and configuration are also part of the virtual machine.

---

# Hypervisor Types

A **hypervisor** is the software layer responsible for creating, running and managing virtual machines.

It provides guest operating systems with access to virtual hardware while controlling how physical hardware resources are shared between multiple virtual machines.

Without a hypervisor, multiple operating systems cannot run simultaneously in isolated virtual machines on the same physical host.

---

## Hypervisor vs. Virtual Machine Monitor (VMM)

The terms **Hypervisor** and **Virtual Machine Monitor (VMM)** are often used interchangeably.

Historically, a **Virtual Machine Monitor (VMM)** referred to the software component responsible for monitoring and managing virtual machines.

Today, the term **hypervisor** is more common and usually describes the same concept.

For LPIC-3, both terms can generally be treated as equivalent unless a question explicitly distinguishes between them.

---

## Type 1 Hypervisors

A **Type 1 Hypervisor**, also called a **bare-metal hypervisor**, runs directly on the physical hardware.

There is no general-purpose host operating system between the hardware and the hypervisor.

Examples include:

- VMware ESXi
- Xen
- Microsoft Hyper-V
- Citrix Hypervisor
- KVM as part of the Linux kernel virtualization stack

Type 1 hypervisors are commonly used in data centers because they provide high performance and efficient resource management.

---

## Type 2 Hypervisors

A **Type 2 Hypervisor** runs as an application on top of a host operating system.

The host operating system is responsible for managing physical hardware and device drivers.

Examples include:

- Oracle VirtualBox
- VMware Workstation
- VMware Fusion
- Parallels Desktop

Type 2 hypervisors are commonly used for development, testing and desktop virtualization.

---

## Comparison

| Feature | Type 1 | Type 2 |
|---------|--------|--------|
| Runs on | Physical hardware | Host operating system |
| Typical use | Servers, data centers | Desktop systems |
| Performance | Higher | Slightly lower |
| Examples | ESXi, Xen, Hyper-V, KVM | VirtualBox, VMware Workstation |

> **Exam Tip**
>
> The difference between Type 1 and Type 2 hypervisors is determined by **where the hypervisor runs**, not by which product is more familiar.

> **Common Mistake**
>
> Do not confuse a **hypervisor** with a **management platform**.
>
> Products such as Proxmox VE or VMware vCenter provide management capabilities, but they are not themselves the low-level virtualization mechanism.

---

# Linux Virtualization Stack

One of the most common misconceptions is assuming that KVM, QEMU, libvirt and Proxmox VE perform the same task.

Each component has a distinct responsibility. Together these components form the Linux virtualization stack.

| Component | Layer | Purpose |
|-----------|-------|---------|
| KVM | Kernel | Hardware-assisted virtualization |
| QEMU | Userspace | Virtual machine process and device emulation |
| libvirt | Management | Virtualization API and daemon |
| Proxmox VE | Management Platform | Management of virtual machines and containers |

```text
                     Proxmox VE
                           │
              +------------+------------+
              │                         │
           Web GUI                  qm CLI
              │                         │
              +----------- API ---------+
                          │
                    QEMU process
                          │
                      /dev/kvm
                          │
                 Linux Kernel + KVM
                          │
                  Intel VT-x / AMD-V
                          │
                      CPU Hardware
```

> **Common Mistake**
>
> KVM, QEMU, libvirt and Proxmox VE are complementary components, not competing products.

# Components

## KVM

**KVM** stands for **Kernel-based Virtual Machine**.

KVM is the virtualization facility built into the Linux kernel. It enables Linux to use hardware-assisted virtualization features provided by modern processors.

Unlike complete virtualization products, KVM focuses on CPU virtualization and works together with QEMU to provide a complete virtual machine.

---

### Kernel Modules

KVM is implemented as Linux kernel modules.

Common modules are:

- `kvm`
- `kvm_intel`
- `kvm_amd`

The CPU-specific module depends on the processor vendor:

- Intel processors use `kvm_intel`
- AMD processors use `kvm_amd`

---

### Hardware-assisted Virtualization

KVM relies on CPU virtualization extensions.

The relevant technologies are:

- **Intel VT-x**
- **AMD-V**

These processor features allow guest operating systems to execute CPU instructions efficiently on the physical processor.

> **Common Mistake**
>
> Do not confuse **Intel VT-x** with **Intel VT-d**.
>
> - **VT-x** provides CPU virtualization.
> - **VT-d** provides I/O virtualization, for example PCI passthrough via the IOMMU.

---

### `/dev/kvm`

KVM exposes its functionality through the device file:

`/dev/kvm`

Applications running in userspace, such as QEMU, use this interface to communicate with the KVM subsystem inside the Linux kernel.

---

### Relationship to QEMU

KVM and QEMU have different responsibilities.

| Component | Responsibility |
|-----------|----------------|
| KVM | Hardware-assisted CPU virtualization |
| QEMU | Virtual machine process and virtual hardware |

```text
QEMU
  │
  ▼
/dev/kvm
  │
  ▼
Linux Kernel + KVM
  │
  ▼
Intel VT-x / AMD-V
```

Without KVM, QEMU can still emulate an entire computer, but performance is significantly lower because CPU instructions must be emulated instead of executed directly by the processor.

---

### Is KVM a Hypervisor?

KVM is commonly regarded as the **hypervisor component** of the Linux virtualization stack.

At first glance, Linux virtualization looks different from a traditional Type 1 hypervisor because the Linux kernel is visible in the architecture.

However, the Linux kernel together with KVM forms the hypervisor layer.

```text
Classic Type 1 Hypervisor

+----------------------+
| Guest Operating Sys. |
+----------------------+
|      Hypervisor      |
+----------------------+
|      Hardware        |
+----------------------+

Linux + KVM

+----------------------+
| Guest Operating Sys. |
+----------------------+
|        QEMU          |
+----------------------+
| Linux Kernel + KVM   |
+----------------------+
|      Hardware        |
+----------------------+
```

> **Exam Tip**
>
> For LPIC-3, KVM can be considered a **Type 1 hypervisor** because virtualization is performed by the Linux kernel directly on the hardware using Intel VT-x or AMD-V.

---

### Verification Commands

Show loaded KVM modules:

```bash
lsmod | grep kvm
```

Check whether the KVM device exists:

```bash
ls -l /dev/kvm
```

Check whether the processor supports hardware virtualization:

```bash
grep -E 'vmx|svm' /proc/cpuinfo
```

Meaning:

- `vmx` = Intel VT-x
- `svm` = AMD-V

## QEMU

**QEMU** stands for **Quick Emulator**.

QEMU is a userspace application that creates and runs virtual machines.

It provides the complete virtual machine process, including virtual hardware such as CPUs, memory, storage controllers, network adapters, firmware and other devices.

When hardware-assisted virtualization is available, QEMU delegates CPU execution to KVM while continuing to manage the virtual machine itself.

---

### Responsibilities

Typical QEMU responsibilities include:

- creating the virtual machine process
- allocating virtual memory
- presenting virtual CPUs
- emulating virtual hardware devices
- providing virtual firmware such as BIOS or UEFI
- attaching virtual disks
- attaching virtual network interfaces
- handling virtual input and output devices

Unlike KVM, QEMU provides the complete virtual hardware environment required by a guest operating system.

---

### Relationship to KVM

```text
                Guest Operating System
                         │
                 Virtual Hardware
                         │
                      QEMU
                         │
                     /dev/kvm
                         │
                 Linux Kernel + KVM
                         │
                    CPU Hardware
```

When KVM is available, guest CPU instructions are executed directly by the physical processor.

Without KVM, QEMU performs software emulation of the processor, which is considerably slower.

---

### Emulation vs. Virtualization in QEMU

QEMU can operate in different modes.

#### QEMU with KVM

- hardware-assisted virtualization
- near-native CPU performance
- commonly used in production environments

#### QEMU without KVM

- complete CPU emulation
- significantly slower
- useful for testing different processor architectures

For example, an x86_64 system can emulate an ARM system using QEMU without requiring ARM hardware.

> **Common Mistake**
>
> QEMU creates and manages the virtual machine.
>
> KVM accelerates CPU execution using hardware-assisted virtualization.

---

### Verification Commands

Check whether QEMU is installed:

```bash
qemu-system-x86_64 --version
```

List available QEMU binaries:

```bash
ls /usr/bin/qemu-*
```

Show running QEMU virtual machines:

```bash
ps -ef | grep qemu
```

---

## libvirt

**libvirt** is an open-source virtualization management framework.

Unlike KVM or QEMU, libvirt does not provide virtualization itself.

Instead, it offers a common management interface for different virtualization technologies.

Applications can use libvirt to create, configure, start, stop and monitor virtual machines without interacting directly with the underlying hypervisor.

---

### Responsibilities

Typical libvirt responsibilities include:

- creating virtual machines
- starting and stopping virtual machines
- configuring virtual hardware
- managing virtual networks
- managing virtual storage pools
- taking snapshots
- monitoring virtual machine status

libvirt communicates with the virtualization backend and exposes a consistent API to management applications.

---

### Relationship to KVM and QEMU

```text
Management Applications
(virt-manager, virsh, OpenStack, etc.)
                │
             libvirt
                │
              QEMU
                │
         Linux Kernel + KVM
                │
            CPU Hardware
```

This layered architecture allows management applications to work without needing to know implementation details of the underlying virtualization technology.

---

### libvirtd

The central component of libvirt is the **libvirtd** daemon.

It listens for requests from management applications and performs the requested operations using the appropriate virtualization backend.

Modern Linux distributions may split libvirt functionality into several specialized daemons, but the LPIC-3 objectives primarily refer to the traditional **libvirtd** service.

---

### Common Tools

| Tool | Purpose |
|------|---------|
| `virsh` | Command-line management tool |
| `virt-manager` | Graphical virtual machine manager |
| `virt-install` | Create new virtual machines |
| `virt-viewer` | Connect to a virtual machine console |

All of these communicate with libvirt instead of interacting directly with QEMU or KVM.

> **Common Mistake**
>
> libvirt is **not** a hypervisor.
>
> It is a management framework that communicates with virtualization backends such as QEMU/KVM.

---

### Verification Commands

```bash
virsh --version
systemctl status libvirtd
virsh list --all
virsh version
```

---

## Proxmox VE

**Proxmox Virtual Environment (Proxmox VE)** is an open-source virtualization management platform.

Unlike KVM, QEMU or libvirt, Proxmox VE is not responsible for low-level virtualization itself.

Instead, it combines multiple virtualization technologies and provides a unified platform for deploying, managing and monitoring virtual machines and containers.

---

### Responsibilities

Typical responsibilities include:

- managing virtual machines
- managing Linux containers (LXC)
- providing a web-based management interface
- cluster management
- live migration
- storage management
- backup and restore
- high availability (HA)

Proxmox VE integrates several open-source components into a single management platform.

---

### Relationship to KVM, QEMU and LXC

```text
+--------------------------------------+
|             Proxmox VE               |
|        Management Platform           |
+--------------------------------------+
|        QEMU / LXC Management         |
+--------------------------------------+
|        Linux Kernel + KVM            |
+--------------------------------------+
|              Hardware                |
+--------------------------------------+
```

For virtual machines, Proxmox VE uses:

- **KVM** for hardware-assisted virtualization
- **QEMU** for the virtual machine process and virtual hardware

For operating-system-level virtualization, Proxmox VE uses:

- **LXC (Linux Containers)**

This allows a single platform to manage both virtual machines and Linux containers.

> **Common Mistake**
>
> Proxmox VE is **not** the hypervisor itself.
>
> It is a management platform built on top of technologies such as KVM, QEMU and LXC.

---

### Verification Commands

```bash
pveversion
pveversion -v
qm list
pct list
```

# Virtualization Techniques

Modern virtualization can be implemented using different techniques.

Historically, these techniques evolved over time as processor capabilities improved.

Today, hardware-assisted virtualization is the standard approach used by most virtualization platforms.

---

## Emulation

**Emulation** means one system imitates another system in software.

An emulator can run software built for a different hardware platform or CPU architecture.

For example, QEMU can emulate an ARM system on an x86_64 host.

Emulation is flexible but usually slower than virtualization because many hardware operations must be translated in software.

---

## Simulation

**Simulation** models the behavior of a system.

A simulator does not necessarily provide a complete executable hardware environment. It may only model selected aspects of a system for testing, analysis or education.

> **Exam Tip**
>
> Emulation aims to imitate another system closely enough to run software.
> Simulation models behavior and does not necessarily execute a real guest operating system.

---

## Hardware-assisted Virtualization

Modern processors provide dedicated virtualization extensions.

Examples include:

- Intel VT-x
- AMD-V

These technologies allow guest operating systems to execute CPU instructions directly on the physical processor while remaining isolated from the host system.

Hardware-assisted virtualization significantly improves performance compared to pure software emulation.

---

## Full Virtualization

**Full Virtualization** allows an unmodified guest operating system to run inside a virtual machine.

The guest operating system does not need to know that it is running in a virtualized environment.

Modern full virtualization typically combines:

- hardware-assisted CPU virtualization
- virtual hardware provided by QEMU or a similar virtual machine monitor

This is the common approach used by KVM/QEMU, VMware ESXi and many other hypervisors.

---

## Hardware Virtual Machine (HVM)

**Hardware Virtual Machine (HVM)** is a term commonly associated with Xen.

An HVM guest uses hardware-assisted virtualization and can usually run an unmodified operating system.

In simple terms:

```text
HVM = full virtualization using CPU hardware support
```

HVM guests rely on technologies such as Intel VT-x or AMD-V.

---

## Paravirtualization (PV)

**Paravirtualization (PV)** requires the guest operating system to be aware that it is running inside a virtual machine.

Instead of behaving exactly as if it were running on physical hardware, the guest cooperates with the hypervisor through specialized interfaces.

Historically, paravirtualization improved performance before modern processors supported efficient hardware-assisted virtualization.

Today, full paravirtualization is less common because Intel VT-x and AMD-V provide efficient hardware support.

However, paravirtualized devices are still widely used.

Examples include:

- VirtIO network devices
- VirtIO block devices
- VirtIO SCSI controllers

> **Common Mistake**
>
> Do not confuse **paravirtualization** with **paravirtualized devices**.
>
> Modern virtual machines often use full virtualization together with paravirtualized drivers such as VirtIO.

---

## Comparison

| Technique | Guest OS modified | Hardware support required | Typical today |
|-----------|-------------------|---------------------------|---------------|
| Emulation | No | No | Special cases |
| Full Virtualization | No | Usually yes | Yes |
| HVM | No | Yes | Yes |
| Paravirtualization | Yes | No | Less common |
| Paravirtualized Devices | Guest drivers | Usually yes | Very common |

# CPU Flags and `/proc/cpuinfo`

CPU virtualization support can be inspected using `/proc/cpuinfo`.

```bash
grep -E 'vmx|svm' /proc/cpuinfo
```

Important flags:

- `vmx` = Intel VT-x
- `svm` = AMD-V

If neither flag is present, hardware-assisted virtualization may not be available, may be disabled in firmware, or may not be exposed to the current virtual machine.

> **Exam Tip**
>
> `/proc/cpuinfo` and CPU flags are explicitly listed in the LPIC-3 objective.

# Virtual Machine Features

Virtualization adds features that are difficult or impossible to provide on physical machines without additional tooling.

---

## Snapshots

A **snapshot** captures the state of a virtual machine at a specific point in time.

Depending on the platform, a snapshot may include:

- virtual disk state
- memory state
- device state
- VM configuration

Snapshots are useful before risky changes, updates or tests.

They are not a replacement for backups.

> **Common Mistake**
>
> Snapshots are not backups.
> They are often dependent on the original VM disks and storage backend.

---

## Pause and Resume

Pausing a virtual machine temporarily stops execution without shutting down the guest operating system.

The VM can later be resumed from the same state.

This is useful for maintenance, debugging or temporary resource management.

---

## Cloning

**Cloning** creates a copy of an existing virtual machine.

Typical use cases include:

- creating test systems
- deploying similar servers
- creating templates
- preparing lab environments

A clone may be a full independent copy or a linked clone depending on the virtualization platform.

---

## Resource Limits

Virtual machines can be assigned or restricted by resource limits.

Examples include:

- number of vCPUs
- amount of RAM
- disk size
- disk I/O limits
- network bandwidth limits
- CPU shares or quotas

Resource limits allow multiple VMs to share a host safely and predictably.

---

# Migration Concepts

Virtualization makes it possible to move workloads between environments more flexibly than with physical machines.

LPIC-3 expects understanding of major migration concepts, not deep command-level usage in this objective.

---

## Physical-to-Virtual Migration (P2V)

**P2V** means migrating a physical machine into a virtual machine.

Typical steps include:

- inventory the physical system
- capture or convert disks
- create a virtual machine with compatible virtual hardware
- adjust drivers and boot configuration
- test the migrated system

Common challenges include:

- hardware-specific drivers
- bootloader issues
- storage controller differences
- network interface changes
- licensing and activation

---

## Virtual-to-Virtual Migration (V2V)

**V2V** means migrating a virtual machine from one virtualization platform or format to another.

Examples include:

- VMware to KVM/QEMU
- VirtualBox to KVM/QEMU
- Xen to KVM/QEMU

Typical tasks include:

- converting disk image formats
- adapting virtual hardware
- changing guest drivers
- testing boot and network configuration

---

## Migration Between Host Systems

Virtual machines can often be moved from one host to another.

Common forms include:

- **cold migration**: the VM is powered off during migration
- **live migration**: the VM continues running while being moved

Live migration typically requires compatible CPUs, shared or replicated storage, network connectivity and compatible virtualization software.

---

## Migration Risks

Important migration considerations include:

- CPU compatibility
- storage access
- network configuration
- downtime requirements
- guest driver support
- performance after migration
- licensing implications

> **Exam Tip**
>
> For 351.1, focus on the meaning and implications of P2V, V2V and host migration rather than exact tool syntax.

# Awareness Topics

The official objective requires awareness of several related platforms and technologies.

This means you should recognize what they are and where they fit, but you do not need deep operational knowledge in this objective.

---

## VirtualBox

**Oracle VirtualBox** is a Type 2 desktop virtualization product.

It runs on top of a host operating system and is commonly used for development, testing and education.

---

## oVirt

**oVirt** is an open-source virtualization management platform for managing KVM-based virtualization environments.

It provides centralized management of hosts, virtual machines, storage and networking.

---

## systemd-machined

**systemd-machined** is a systemd service that tracks local virtual machines and containers.

It is part of the systemd machine management infrastructure and is used by tools such as `machinectl`.

---

## Open vSwitch

**Open vSwitch (OVS)** is a multilayer virtual switch commonly used in virtualization and cloud environments.

It can provide advanced virtual networking features such as bridges, VLANs, tunnels and software-defined networking integration.

> **Exam Tip**
>
> LPIC-3 351.1 expects awareness of Open vSwitch, not deep Open vSwitch administration.

# Summary

Virtualization allows multiple isolated operating systems to run on the same physical hardware.

A hypervisor or VMM provides and controls the virtual machine environment.

Type 1 hypervisors run directly on hardware, while Type 2 hypervisors run on top of a host operating system.

On Linux, KVM provides hardware-assisted CPU virtualization in the kernel, QEMU provides the virtual machine process and virtual hardware, libvirt provides a management API, and Proxmox VE provides a complete management platform.

Modern virtualization usually combines full virtualization with hardware-assisted CPU features such as Intel VT-x or AMD-V.

Paravirtualization requires guest cooperation, while paravirtualized devices such as VirtIO remain common for performance.

LPIC-3 also expects understanding of snapshots, pausing, cloning, resource limits, P2V, V2V, host migration, `/proc/cpuinfo`, CPU flags and awareness of related technologies such as VirtualBox, oVirt, systemd-machined and Open vSwitch.
