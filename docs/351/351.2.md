# Xen Overview

**Xen** is a Type 1 hypervisor.

It runs directly on the hardware and controls CPU, memory and interrupt virtualization for guest systems.

Unlike a normal Linux host running only user-space virtual machine processes, a Xen system boots the Xen hypervisor first. A privileged management domain, called **Domain0** or **Dom0**, is then started by the hypervisor.

Dom0 provides the administrative environment used to create and manage other virtual machines.

Those virtual machines are called **DomainU** or **DomU** guests.

```text
+------------------------------------------------+
| DomU guest      | DomU guest      | DomU guest |
| PV or HVM       | PV or HVM       | PV or HVM  |
+-----------------+-----------------+------------+
|                Dom0 Linux system               |
|     xl, xenstored, backend drivers, tools      |
+------------------------------------------------+
|                 Xen hypervisor                 |
+------------------------------------------------+
|                    Hardware                    |
+------------------------------------------------+
```

> **Exam Tip**
>
> In Xen terminology, remember the domain names:
>
> - **Dom0** is the privileged control domain.
> - **DomU** is an unprivileged guest domain.

# Xen Architecture

## Nodes and Domains

A **Xen node** is a host system running the Xen hypervisor.

A **Xen domain** is a virtual machine environment running on the Xen hypervisor.

Dom0 and DomU are both domains, but they have different privileges.

- **Dom0** is the privileged control domain.
- **DomU** is an unprivileged guest domain.

For LPIC-3, node-level tasks usually refer to the Xen host, while domain-level tasks refer to guest virtual machines and their configuration.

---

## Xen Hypervisor

The Xen hypervisor is the low-level virtualization layer.

It is responsible for:

- scheduling virtual CPUs on physical CPUs
- managing memory isolation
- handling privileged CPU operations
- routing interrupts
- enforcing isolation between domains

The hypervisor itself is intentionally small.

Most device drivers and management tools run outside the hypervisor, usually in Dom0.

---

## Domain0

**Domain0**, usually written as **Dom0**, is the first domain started by Xen.

Dom0 is privileged and has direct or controlled access to physical hardware.

Dom0 typically provides:

- Xen management tools such as `xl`
- access to domain configuration files
- backend drivers for block and network devices
- access to physical storage and network interfaces
- log files and troubleshooting tools

Dom0 is often a Linux system with Xen packages installed.

> **Common Mistake**
>
> Dom0 is not the Xen hypervisor itself.
>
> Xen is the hypervisor. Dom0 is a privileged operating system running on top of Xen.

---

## DomainU

**DomainU**, usually written as **DomU**, refers to an unprivileged guest domain.

A DomU is a virtual machine managed by Xen.

DomU guests can run in different virtualization modes, most importantly:

- PV-DomU
- HVM-DomU

DomU guests do not directly manage the physical host.

They use virtual devices provided by Xen and backend services in Dom0.

---

## PV-DomU

**PV-DomU** means **paravirtualized DomainU**.

A PV guest is aware that it is running under Xen and uses Xen-specific interfaces instead of pretending that all hardware is physical.

Historically, PV guests were important because they allowed efficient virtualization before modern hardware virtualization extensions became common.

PV guests usually require a Xen-aware operating system kernel.

---

## HVM-DomU

**HVM-DomU** means **Hardware Virtual Machine DomainU**.

An HVM guest uses hardware-assisted virtualization features such as Intel VT-x or AMD-V.

HVM allows Xen to run operating systems that do not need to be modified for Xen.

Modern Xen deployments commonly use HVM guests together with paravirtualized drivers for better disk and network performance.

> **Exam Tip**
>
> A useful exam distinction:
>
> - **PV-DomU**: guest cooperates with Xen.
> - **HVM-DomU**: guest can run as a hardware-assisted virtual machine.

---

# Xen Toolstack

The Xen hypervisor needs management tools.

The main tool for Xen 4.x administration is `xl`.

`xl` uses the **libxl** toolstack library to manage domains.

Most `xl` commands require root privileges because they communicate with the Xen hypervisor and Xen control services.

Common responsibilities include:

- showing host information
- listing domains
- creating domains from configuration files
- shutting down or destroying domains
- connecting to guest consoles
- migrating domains
- adjusting virtual CPUs and memory

```text
administrator
     |
     v
    xl
     |
     v
  libxl
     |
     v
 Xen hypervisor + xenstored + Dom0 services
```

# Files and Directories

## `/etc/xen/`

`/etc/xen/` is the traditional Xen configuration directory.

It commonly contains:

- global Xen toolstack configuration
- domain configuration files
- helper scripts
- example configuration files

The exact contents are distribution-dependent.

For LPIC-3, recognize `/etc/xen/` as the central Xen configuration location.

---

## `xl.conf`

`xl.conf` is the global configuration file for the `xl` toolstack.

It can define default behavior used by `xl`.

The usual path is:

```text
/etc/xen/xl.conf
```

Examples of settings that may appear in `xl.conf` include default backend behavior, console behavior or other toolstack defaults.

> **Exam Tip**
>
> `xl.conf` configures the `xl` toolstack.
> It is not the per-guest virtual machine configuration file.

---

## `xl.cfg`

`xl.cfg` refers to the Xen domain configuration format used by `xl`.

A domain configuration file describes one guest domain.

It defines settings such as:

- domain name
- memory
- virtual CPUs
- virtualization type
- disk devices
- network interfaces
- boot behavior
- console behavior

Domain configuration files are often stored below `/etc/xen/`, for example:

```text
/etc/xen/debian.cfg
/etc/xen/web01.cfg
/etc/xen/testvm.cfg
```

---

# Basic Domain Configuration

The exact syntax supported by Xen configuration files can vary between Xen versions and distributions.

For LPIC-3, focus on recognizing the purpose of common options rather than memorizing every possible form.

## Minimal PV-style Example

```ini
name = "pvguest1"
memory = 1024
vcpus = 2
type = "pv"

kernel = "/boot/vmlinuz"
ramdisk = "/boot/initrd.img"
extra = "root=/dev/xvda1 ro"

disk = [
    "format=raw, vdev=xvda, access=rw, target=/var/lib/xen/images/pvguest1.img"
]

vif = [
    "bridge=xenbr0"
]
```

Important options:

| Option | Meaning |
|--------|---------|
| `name` | Domain name |
| `memory` | Guest memory in MiB |
| `vcpus` | Number of virtual CPUs |
| `type = "pv"` | Defines a PV guest |
| `kernel` | Kernel used to boot a PV guest |
| `ramdisk` | Initial RAM disk |
| `extra` | Additional kernel command-line parameters |
| `disk` | Guest block devices |
| `vif` | Guest virtual network interfaces |

---

## Minimal HVM-style Example

```ini
name = "hvmguest1"
memory = 2048
vcpus = 2
type = "hvm"

disk = [
    "format=raw, vdev=hda, access=rw, target=/var/lib/xen/images/hvmguest1.img",
    "format=raw, vdev=hdc, access=ro, devtype=cdrom, target=/var/lib/xen/iso/installer.iso"
]

vif = [
    "bridge=xenbr0"
]

boot = "dc"
vnc = 1
```

Important options:

| Option | Meaning |
|--------|---------|
| `type = "hvm"` | Defines an HVM guest |
| `boot` | Boot order for HVM guests |
| `vnc` | Enables VNC display access when supported |
| `disk` | Defines virtual disks or installation media |
| `vif` | Defines virtual network interfaces |

> **Exam Tip**
>
> Older Xen examples may use disk strings such as `file:/path/to/disk.img,xvda,w` or `phy:/dev/vg0/web01,xvda,w`.
> Current `xl` documentation prefers the more explicit `format=`, `vdev=`, `access=` and `target=` form.

> **Common Mistake**
>
> Do not assume every Xen guest uses the same boot method.
>
> PV guests may boot using a kernel and initrd specified by Dom0.
> HVM guests usually boot more like physical machines, using virtual firmware and boot devices.

# Xen Networking

Xen guests use virtual network interfaces.

A guest network device is usually connected to a backend network device in Dom0.

The common model is:

```text
DomU eth0
   |
virtual interface
   |
Dom0 backend interface
   |
Linux bridge or virtual switch
   |
physical NIC
   |
external network
```

---

## Bridged Networking

Bridged networking connects guest interfaces to a bridge in Dom0.

The bridge behaves like a virtual Ethernet switch.

If the bridge is connected to a physical interface, guests can appear on the same Layer 2 network as the host.

Example domain configuration:

```ini
vif = [
    "bridge=xenbr0"
]
```

Common commands used to inspect networking include:

```bash
ip link show
ip addr show
bridge link
```

Depending on the distribution, the bridge may be configured by system networking tools rather than Xen itself.

---

## NAT and Routed Networking

Some Xen installations use NAT or routed networking instead of bridging.

In these designs, Dom0 routes or translates traffic between guest networks and external networks.

For the LPIC-3 objective, the important point is to understand that Xen networking depends on Dom0 network configuration and guest virtual interfaces.

---

## Network Troubleshooting

When a Xen guest has no network connectivity, check:

- whether the guest has a virtual interface
- whether the domain configuration has a valid `vif` entry
- whether the Dom0 bridge exists
- whether the physical interface is connected to the bridge
- whether firewall rules or routing rules block traffic
- whether the guest operating system configured its interface correctly

Useful commands:

```bash
xl network-list DOMAIN
ip link show
bridge link
ip route show
```

---

# Xen Storage

Xen presents virtual block devices to guest domains.

The underlying storage can come from different sources, including:

- raw image files
- block devices
- LVM logical volumes
- network storage
- storage managed by a larger platform

Xen itself does not require one specific storage backend.

The guest sees a virtual disk, while Dom0 maps that virtual disk to real storage.

---

## Disk Configuration

A common `disk` entry has several important pieces:

```ini
disk = [
    "format=raw, vdev=xvda, access=rw, target=/var/lib/xen/images/web01.img"
]
```

This can be read as:

| Part | Meaning |
|------|---------|
| `format=raw` | Disk image format |
| `vdev=xvda` | Device name exposed to the guest |
| `access=rw` | Guest has read-write access |
| `target=/var/lib/xen/images/web01.img` | Storage backend in Dom0 |

Another common form uses a block device:

```ini
disk = [
    "format=raw, vdev=xvda, access=rw, target=/dev/vg0/web01"
]
```

Here, `target=/dev/vg0/web01` indicates that the backend is a block device or logical volume.

---

## Storage Troubleshooting

When a Xen guest cannot access a disk, check:

- whether the image file or block device exists
- whether Dom0 has permission to access the backend
- whether the disk syntax in the domain configuration is correct
- whether the target guest device name is correct
- whether the backend storage is already in use in an unsafe way
- whether Dom0 logs contain backend driver errors

Useful commands:

```bash
ls -l /var/lib/xen/images/
lvs
xl block-list DOMAIN
xl dmesg
dmesg
```

# Basic Xen Installation Concepts

The official objective includes installing and maintaining Xen installations.

Package names and bootloader integration vary by Linux distribution, but the high-level process is usually:

1. Install the Xen hypervisor and tools.
2. Install or select a Dom0-capable Linux kernel.
3. Configure the bootloader to start Xen before the Dom0 kernel.
4. Reboot into the Xen hypervisor.
5. Verify Xen with `xl info`.
6. Verify that Xen support services are running.
7. Configure storage and networking for guest domains.

After a successful Xen boot, the system is no longer just a normal Linux host.

It is a Xen host with a Dom0 Linux system running under the Xen hypervisor.

---

## Verifying Xen

Use `xl info` to verify that the Xen toolstack can communicate with the hypervisor:

```bash
xl info
```

Useful fields include:

| Field | Meaning |
|-------|---------|
| `host` | Hostname |
| `release` | Dom0 kernel release |
| `xen_version` | Xen hypervisor version |
| `nr_cpus` | Available physical CPU threads |
| `total_memory` | Host memory visible to Xen |
| `free_memory` | Memory currently free for domains |

If `xl info` cannot connect to Xen, the system may not have booted into the Xen hypervisor or required Xen services may not be running.

On many distributions, Xen support services are started through a service such as `xencommons`.
These services commonly include components such as `xenstored` and `xenconsoled`, which are needed by the Xen toolstack.

# Managing Domains with `xl`

## Listing Domains

```bash
xl list
```

Example output:

```text
Name                                        ID   Mem VCPUs      State   Time(s)
Domain-0                                     0  4096     4     r-----     120.4
web01                                        3  2048     2     -b----      35.7
```

Important points:

- `Domain-0` is Dom0.
- Other entries are DomU guests.
- The `State` column shows whether the domain is running, blocked, paused or otherwise inactive.

Common state letters include:

| Letter | Meaning |
|--------|---------|
| `r` | running |
| `b` | blocked, usually waiting for I/O |
| `p` | paused |
| `s` | shutdown |
| `c` | crashed |
| `d` | dying |

---

## Creating a Domain

Create a domain from a configuration file:

```bash
xl create /etc/xen/web01.cfg
```

Create a domain and attach to its console:

```bash
xl create -c /etc/xen/web01.cfg
```

The configuration file defines memory, CPUs, disks, network interfaces and boot behavior.

The examples use absolute paths to make the configuration file location explicit.

---

## Console Access

Attach to a guest console:

```bash
xl console web01
```

To leave many Xen consoles, use the console escape sequence:

```text
Ctrl+]
```

> **Exam Tip**
>
> `xl console` connects to a guest console.
> It is not the same as SSH access into the guest operating system.

---

## Graceful Shutdown and Reboot

Ask a guest to shut down:

```bash
xl shutdown web01
```

Ask a guest to reboot:

```bash
xl reboot web01
```

These commands depend on guest cooperation.

If the guest operating system is hung or cannot respond, the command may not complete as expected.

---

## Destroying a Domain

Forcefully stop a domain:

```bash
xl destroy web01
```

`xl destroy` is comparable to pulling power from a physical machine.

It should not be the normal shutdown method for a healthy guest.

> **Common Mistake**
>
> `xl shutdown` requests a clean shutdown.
> `xl destroy` forcefully terminates the domain.

---

## Pausing and Unpausing

Pause a domain:

```bash
xl pause web01
```

Resume a paused domain:

```bash
xl unpause web01
```

Pausing stops guest execution without removing the domain.

---

## Virtual CPU Management

List virtual CPUs:

```bash
xl vcpu-list
```

Set the number of active virtual CPUs for a domain:

```bash
xl vcpu-set web01 2
```

The guest operating system and domain configuration must support the requested CPU layout.

---

## Memory Management

Show domain memory information:

```bash
xl list
```

Adjust domain memory:

```bash
xl mem-set web01 2048m
```

Memory changes depend on guest support and Xen configuration.
Use an explicit unit such as `m` or `g` to avoid ambiguity.

For exam purposes, understand that Xen can control memory assigned to domains and that Dom0 memory must also be planned.

---

## Monitoring with `xentop`

`xentop` is an interactive monitoring tool for Xen.

It shows resource usage for Xen domains.

```bash
xentop
```

Typical information includes:

- domain name
- domain state
- CPU usage
- memory usage
- virtual CPUs
- network and block I/O statistics

`xentop` is useful for quickly identifying busy or resource-constrained domains.

# Xen Migration

Xen can migrate domains between hosts when the environment supports it.

Migration usually requires:

- compatible Xen versions
- compatible CPU features
- network connectivity between hosts
- suitable storage access
- enough memory and CPU capacity on the target host
- compatible domain configuration

The basic command form is:

```bash
xl migrate DOMAIN TARGET_HOST
```

Depending on the environment, migration may be live or non-live.

Live migration attempts to keep the guest running while memory state is transferred to the target host.

> **Exam Tip**
>
> For 351.2, focus on the concept and requirements of Xen migration.
> Deep cluster design and storage architecture are beyond this objective.

# Xen Boot Parameters

Xen boot parameters are options passed to the Xen hypervisor at boot time.

They are separate from Linux kernel parameters passed to Dom0.

In a bootloader configuration, this distinction may look conceptually like this:

```text
Xen hypervisor line: xen.gz dom0_mem=4096M
Dom0 kernel line:    vmlinuz root=/dev/mapper/vg-root ro
```

Common reasons to use Xen boot parameters include:

- reserving memory for Dom0
- limiting Dom0 virtual CPUs
- enabling or changing IOMMU behavior
- changing hypervisor logging
- adjusting CPU or scheduler behavior

Commonly seen examples include:

| Parameter | Purpose |
|-----------|---------|
| `dom0_mem=` | Set or limit Dom0 memory |
| `dom0_max_vcpus=` | Limit the number of Dom0 virtual CPUs |
| `iommu=` | Configure IOMMU behavior |
| `loglvl=` | Adjust Xen hypervisor log verbosity |

> **Exam Tip**
>
> Know that Xen boot parameters configure the Xen hypervisor.
> They are not the same thing as a guest domain's `extra` kernel parameters in an `xl.cfg` file.

---

# XenStore

**XenStore** is a shared configuration and status database used by Xen.

It stores information used by the hypervisor, Dom0 and DomU guests.

XenStore is commonly used for:

- domain state information
- virtual device configuration
- frontend and backend driver coordination
- guest and toolstack communication

The XenStore service is usually provided by `xenstored`.

For LPIC-3, you should recognize XenStore as a Xen-specific key-value store used for runtime configuration and communication between Xen components.

---

# XAPI

**XAPI** is a Xen management API and toolstack associated with XenServer-style environments and related platforms.

It provides higher-level management functionality than direct `xl` usage.

XAPI is used for managing hosts, virtual machines, storage and networking through an API-driven management layer.

For LPIC-3 351.2, XAPI is an awareness topic.

You should recognize that XAPI is a Xen management API, not the Xen hypervisor itself.

---

# The Legacy `xm` Utility

`xm` is the older Xen management command associated with the older `xend` toolstack.

Modern Xen 4.x administration focuses on `xl`.

For LPIC-3, you should be aware that:

- `xm` is legacy
- `xl` is the modern command-line tool for Xen 4.x administration
- old documentation may still mention `xm`

> **Common Mistake**
>
> Do not learn `xm` as the main command for current Xen administration.
> Recognize it as an older utility and focus on `xl`.

# Troubleshooting Xen

Troubleshooting Xen usually means checking the correct layer.

The problem may be in:

- bootloader configuration
- Xen hypervisor startup
- Dom0 kernel or services
- toolstack communication
- domain configuration
- storage backend
- network backend
- guest operating system

---

## Host Did Not Boot into Xen

Symptoms:

- `xl info` fails
- no Xen version is shown
- Dom0 appears to be a normal Linux system

Check:

```bash
xl info
dmesg
cat /proc/cmdline
```

Also check the bootloader entry.

The system must boot the Xen hypervisor first, then the Dom0 kernel.

---

## Domain Does Not Start

Symptoms:

- `xl create` fails
- domain exits immediately
- console shows boot errors

Check:

```bash
xl create -c /etc/xen/web01.cfg
xl dmesg
dmesg
```

Common causes:

- invalid domain configuration syntax
- missing kernel or initrd for PV guests
- missing disk image or block device
- insufficient free memory
- incorrect disk device name
- network bridge does not exist

---

## No Guest Console

Symptoms:

- `xl console` connects but shows no output
- guest appears to boot but cannot be accessed

Check:

- guest console settings
- PV kernel console parameters
- HVM display or serial console configuration
- whether the guest is actually running

Useful commands:

```bash
xl list
xl console DOMAIN
```

---

## Guest Has No Network

Symptoms:

- guest interface is missing
- guest cannot reach the network
- Dom0 bridge exists but traffic does not pass

Check:

```bash
xl network-list DOMAIN
ip link show
bridge link
ip addr show
ip route show
```

Common causes:

- wrong bridge name in `vif`
- bridge not configured in Dom0
- physical interface not attached to bridge
- firewall or routing problem
- guest OS network misconfiguration

---

## Guest Disk Is Missing

Symptoms:

- guest cannot find its root disk
- boot fails with storage errors
- `xl create` reports a block device problem

Check:

```bash
xl block-list DOMAIN
ls -l /var/lib/xen/images/
lvs
dmesg
```

Common causes:

- missing image file
- wrong `disk` entry
- wrong guest device name
- permission problem in Dom0
- unavailable LVM volume or storage backend

---

## Resource Problems

Symptoms:

- new domains cannot start
- domains are slow
- host appears overloaded

Check:

```bash
xl info
xl list
xentop
xl vcpu-list
```

Common causes:

- insufficient free memory
- too many vCPUs assigned
- storage bottleneck
- network bottleneck
- Dom0 underprovisioned

# Summary

Xen is a Type 1 hypervisor that boots before the Dom0 operating system.

Dom0 is the privileged control domain used to manage the Xen host and provide backend access to hardware resources.

DomU guests are unprivileged virtual machines. They may run as PV-DomU guests or HVM-DomU guests.

The main Xen 4.x command-line tool is `xl`. It manages domains through the libxl toolstack.

Xen configuration commonly lives under `/etc/xen/`. `xl.conf` configures the `xl` toolstack, while `xl.cfg` domain configuration files define guest settings such as memory, vCPUs, disks and virtual network interfaces.

Xen networking commonly uses virtual interfaces connected to bridges in Dom0. Xen storage maps guest block devices to files, physical block devices, LVM volumes or other backends.

Important commands include `xl info`, `xl list`, `xl create`, `xl console`, `xl shutdown`, `xl destroy`, `xl migrate`, `xl dmesg` and `xentop`.

LPIC-3 also expects awareness of XAPI, XenStore, Xen boot parameters and the legacy `xm` utility.
