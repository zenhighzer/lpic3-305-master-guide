# QEMU Overview

**QEMU** is a machine emulator and virtualizer.

In the LPIC-3 virtualization objectives, QEMU is most important as the userspace program that creates and runs virtual machine processes.

When used with KVM, QEMU provides the virtual machine process and virtual hardware, while KVM provides hardware-assisted CPU virtualization in the Linux kernel.

```text
                 Management Tool
             libvirt, virt-manager, scripts
                         |
                         v
              qemu-system-x86_64 process
                         |
        +----------------+----------------+
        |                                 |
        v                                 v
   Virtual devices                  /dev/kvm
 disk, NIC, display                     |
                                        v
                               Linux kernel + KVM
                                        |
                                        v
                              Intel VT-x / AMD-V
```

> **Exam Tip**
>
> QEMU and KVM are not the same component.
>
> - **QEMU** runs the virtual machine process and provides virtual devices.
> - **KVM** accelerates guest CPU execution in the Linux kernel.

# QEMU, KVM and libvirt

## QEMU

QEMU can emulate complete machines.

It can provide:

- virtual CPUs
- virtual memory
- virtual storage controllers and disks
- virtual network cards
- virtual firmware
- virtual display and input devices
- monitor interfaces for management

QEMU can run without KVM by using software emulation.

On Linux virtualization hosts, QEMU is usually combined with KVM for performance.

---

## KVM

**KVM** stands for **Kernel-based Virtual Machine**.

KVM is the Linux kernel virtualization facility that uses CPU virtualization extensions such as Intel VT-x or AMD-V.

The relevant kernel modules are:

| Module | Purpose |
|--------|---------|
| `kvm` | Common KVM kernel module |
| `kvm-intel` | Intel-specific KVM module |
| `kvm-amd` | AMD-specific KVM module |

Only one vendor-specific module is normally loaded, depending on the CPU vendor.

---

## `/dev/kvm`

`/dev/kvm` is the device file used by userspace programs such as QEMU to access KVM.

If `/dev/kvm` exists and permissions allow access, QEMU can use KVM acceleration.

If `/dev/kvm` is missing or inaccessible, QEMU may fail to start with KVM enabled or may fall back to slower software emulation when configured to do so.

Useful checks:

```bash
lsmod | grep kvm
ls -l /dev/kvm
grep -E 'vmx|svm' /proc/cpuinfo
```

> **Common Mistake**
>
> CPU flags alone are not enough.
>
> For QEMU with KVM, the CPU must support virtualization, the feature must be enabled in firmware, the KVM modules must be loaded, and QEMU must be allowed to access `/dev/kvm`.

---

## libvirt

**libvirt** is a management layer.

It can start and manage QEMU virtual machines, but it is not required to understand basic QEMU command-line operation in this objective.

The next objectives cover libvirt in more detail.

For 351.3, focus on direct QEMU concepts and commands.

# QEMU Commands and Binaries

## `qemu-system-x86_64`

`qemu-system-x86_64` is the QEMU system emulator for x86_64 virtual machines.

It is the main command used in this objective.

The basic command form is:

```bash
qemu-system-x86_64 [options] [disk_image]
```

Modern QEMU uses architecture-specific binaries:

| Binary | Purpose |
|--------|---------|
| `qemu-system-x86_64` | Run x86_64 system virtual machines |
| `qemu-system-aarch64` | Run AArch64 system virtual machines |
| `qemu-system-s390x` | Run IBM Z system virtual machines |

The exact set of binaries depends on installed packages.

---

## `qemu`

The term `qemu` may refer to the QEMU project, the installed QEMU package, or older/generic QEMU commands on some systems.

For LPIC-3, recognize `qemu` as the QEMU virtualization and emulation tool family.

When starting an x86_64 virtual machine directly, expect to see `qemu-system-x86_64`.

# Starting QEMU Instances

## Minimal Command

A minimal QEMU command can boot from a disk image:

```bash
qemu-system-x86_64 disk.img
```

This is simple, but it relies on many defaults.

For exam and troubleshooting purposes, it is better to understand explicit options.

---

## Basic KVM-Accelerated VM

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -m 2048 \
    -smp 2 \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -netdev user,id=net0 \
    -device virtio-net-pci,netdev=net0
```

Important options:

| Option | Meaning |
|--------|---------|
| `-enable-kvm` | Use KVM acceleration |
| `-m 2048` | Allocate 2048 MiB of guest memory |
| `-smp 2` | Provide two virtual CPUs |
| `-drive file=...,format=...,if=...` | Attach a virtual disk |
| `-netdev user,id=net0` | Create a user-mode network backend |
| `-device virtio-net-pci,netdev=net0` | Add a VirtIO network device connected to `net0` |

> **Exam Tip**
>
> `-enable-kvm` is a shortcut for enabling KVM acceleration.
> You may also see `-machine accel=kvm`.

---

## Booting from Installation Media

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -m 2048 \
    -smp 2 \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -cdrom installer.iso \
    -boot d
```

Important options:

| Option | Meaning |
|--------|---------|
| `-cdrom installer.iso` | Attach ISO image as a CD-ROM |
| `-boot d` | Boot from CD-ROM first |

This is a common installation pattern.

---

## Headless QEMU

QEMU can run without a graphical display.

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -m 1024 \
    -drive file=server.qcow2,format=qcow2,if=virtio \
    -nographic
```

`-nographic` redirects guest serial output and monitor interaction to the terminal when the guest is configured for serial console output.

> **Common Mistake**
>
> `-nographic` does not magically make every guest show a login prompt.
> The guest operating system must be configured to use a serial console.

# Important QEMU Configuration Parameters

QEMU is often configured directly on the command line.

The command line is therefore the most important configuration surface for this objective.

| Option | Purpose |
|--------|---------|
| `-enable-kvm` | Enable KVM acceleration |
| `-machine accel=kvm` | Select KVM as the accelerator |
| `-cpu host` | Expose host CPU features to the guest when using KVM |
| `-m SIZE` | Set guest memory |
| `-smp N` | Set virtual CPU count |
| `-drive ...` | Attach block storage |
| `-device ...` | Add a virtual device |
| `-netdev ...` | Define a network backend |
| `-cdrom ISO` | Attach an ISO as CD-ROM media |
| `-boot ORDER` | Set boot order |
| `-monitor ...` | Configure QEMU monitor access |
| `-serial ...` | Configure serial device redirection |
| `-display ...` | Configure graphical display backend |
| `-nographic` | Run without graphical output |
| `-name NAME` | Set VM name |
| `-daemonize` | Run QEMU in the background |
| `-pidfile FILE` | Write QEMU process ID to a file |

> **Exam Tip**
>
> LPIC-3 does not require memorizing every QEMU option.
> Focus on recognizing what common options control: acceleration, CPU, memory, storage, networking, boot, display and monitor access.

# QEMU Architecture

## Process Model

Each directly started QEMU virtual machine is a host process.

You can usually see it with standard process tools:

```bash
ps aux | grep qemu
pgrep -a qemu
```

The QEMU process owns the VM's virtual devices and communicates with the kernel through interfaces such as `/dev/kvm`, TAP devices and file descriptors for disk images.

---

## Acceleration: KVM and TCG

QEMU can execute guest CPU instructions in different ways.

The two important modes for this objective are:

| Mode | Meaning | Performance |
|------|---------|-------------|
| KVM | Hardware-assisted virtualization through `/dev/kvm` | Fast |
| TCG | Tiny Code Generator software emulation | Slower |

**TCG** allows QEMU to emulate CPUs in software.

This is flexible, but it is much slower than KVM for normal virtualization workloads.

For Linux server virtualization on x86_64, QEMU is usually used with KVM.

---

## Storage Architecture

QEMU maps guest block devices to host storage backends.

The backend can be:

- a disk image file
- a raw block device
- a logical volume
- network storage
- a backend managed by another layer

The guest sees a virtual disk controller and virtual disk.

QEMU maps guest I/O to the configured host backend.

```text
Guest OS
   |
VirtIO, IDE, SATA or SCSI disk device
   |
QEMU block layer
   |
qcow2/raw image, block device or network backend
   |
Host filesystem or storage stack
```

---

## Networking Architecture

QEMU networking has two sides:

- a **guest network device**, such as `virtio-net-pci` or an emulated Intel card
- a **host network backend**, such as user-mode networking or TAP

These are connected using an ID.

Example:

```bash
-netdev user,id=net0
-device virtio-net-pci,netdev=net0
```

Here:

- `net0` is the host-side network backend
- `virtio-net-pci` is the guest-visible network card
- `netdev=net0` connects the virtual NIC to the backend

---

# QEMU Storage

## Basic Disk Attachment

The `-drive` option attaches storage to a VM.

Example:

```bash
-drive file=vm.qcow2,format=qcow2,if=virtio
```

Important fields:

| Field | Meaning |
|-------|---------|
| `file=vm.qcow2` | Host disk image file |
| `format=qcow2` | Disk image format |
| `if=virtio` | Expose disk through a VirtIO block device |

`format=` should be explicit.

If QEMU has to guess the image format, startup may produce warnings and behavior can be less predictable.

---

## Raw Block Device Example

QEMU can use block devices directly.

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -drive file=/dev/vg0/vm01,format=raw,if=virtio
```

This requires appropriate host permissions and care.

Writing to the wrong block device can destroy host data.

---

## Storage Troubleshooting

When a QEMU VM cannot boot from disk, check:

- whether the image file exists
- whether the QEMU process has permission to read and write the file
- whether the image format is correct
- whether the guest has the required driver for the virtual controller
- whether the boot order points to the expected device
- whether the host filesystem has free space

Useful commands:

```bash
ls -lh vm.qcow2
file vm.qcow2
df -h
ps aux | grep qemu
```

> **Boundary**
>
> Detailed disk image conversion and inspection with `qemu-img` and libguestfish belongs to objective 351.5.
> In 351.3, focus on attaching and troubleshooting storage used by QEMU.

# QEMU Networking

## User-Mode Networking

User-mode networking is the simplest QEMU networking mode.

It does not require root privileges or host TAP/bridge configuration.

Example:

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -netdev user,id=net0 \
    -device virtio-net-pci,netdev=net0
```

In this mode, QEMU provides a user-space network stack with DHCP and outbound connectivity.

The guest commonly receives an address in the `10.0.2.0/24` range.

Typical properties:

- simple to use
- no host bridge required
- guest can usually reach outside networks
- incoming access to the guest requires port forwarding
- not equivalent to a full bridged network

Example SSH port forwarding:

```bash
-netdev user,id=net0,hostfwd=tcp::2222-:22
```

This forwards host TCP port `2222` to guest TCP port `22`.

---

## TAP Networking

A **TAP** device is a virtual Ethernet device on the host.

QEMU can connect a guest NIC to a TAP device, allowing the guest to participate in host networking more directly.

Example:

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -netdev tap,id=net0,ifname=tap0,script=no,downscript=no \
    -device virtio-net-pci,netdev=net0
```

This expects `tap0` to already exist and be configured.

TAP networking also depends on host TUN/TAP support.
On Linux hosts, `/dev/net/tun` must exist for TAP interfaces to be created.

---

## Bridge Networking

Bridge networking connects the TAP interface to a Linux bridge.

The guest can then appear on the same Layer 2 network as the bridge.

```text
Guest virtio-net
       |
QEMU process
       |
tap0
       |
Linux bridge br0
       |
Physical NIC
       |
External network
```

Modern systems commonly use `ip` and `bridge` commands.

Older documentation may use `brctl` and `tunctl`, which are explicitly listed by the LPIC objective.

---

## `ip`

`ip` is the modern Linux network administration command.

Examples:

```bash
ip link show
ip addr show
ip tuntap add dev tap0 mode tap
ip link set tap0 up
ip link set tap0 master br0
```

---

## `brctl`

`brctl` is the older bridge management command from the `bridge-utils` package.

Examples:

```bash
brctl show
brctl addbr br0
brctl addif br0 tap0
```

Many modern distributions prefer `ip link` and `bridge`, but `brctl` remains exam-relevant because it appears in the objective.

---

## `tunctl`

`tunctl` is an older tool for creating persistent TUN/TAP interfaces.

Example:

```bash
tunctl -t tap0
```

Modern systems often use:

```bash
ip tuntap add dev tap0 mode tap
```

> **Exam Tip**
>
> Know what `ip`, `brctl` and `tunctl` are used for in QEMU networking.
> Do not confuse the guest NIC with the host TAP device or bridge.

---

## Network Troubleshooting

When a QEMU guest has no network connectivity, check:

- whether QEMU has a network backend
- whether the guest has a network device
- whether the backend ID matches the `netdev=` setting
- whether `/dev/net/tun` exists when TAP networking is used
- whether a TAP device exists when TAP networking is used
- whether the TAP device is up
- whether the TAP device is attached to the correct bridge
- whether the guest has an IP address
- whether host firewall or forwarding rules block traffic

Useful commands:

```bash
ip link show
ip addr show
bridge link
brctl show
ps aux | grep qemu
```

In the QEMU monitor:

```text
(qemu) info network
```

# QEMU Monitor

The **QEMU monitor** is an interactive management interface for a running QEMU instance.

It can be used to:

- inspect VM state
- stop and continue execution
- manage removable media
- create and restore VM snapshots
- inspect block and network state
- start migration
- quit the emulator

QEMU has both a human monitor and a machine protocol.

For this objective, **QEMU monitor** usually means the human monitor, often called **HMP**.

---

## Opening a Monitor

Start QEMU with a monitor on standard input/output:

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -m 1024 \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -monitor stdio
```

Inside the monitor, commands appear at a prompt:

```text
(qemu)
```

Common monitor commands:

| Command | Purpose |
|---------|---------|
| `help` | Show monitor help |
| `info version` | Show QEMU version |
| `info status` | Show VM status |
| `info kvm` | Show KVM information |
| `info network` | Show network state |
| `info block` | Show block devices |
| `info snapshots` | Show VM snapshots |
| `stop` | Pause VM execution |
| `cont` | Continue VM execution |
| `quit` | Quit QEMU |

---

# Snapshots with the QEMU Monitor

The official objective specifically mentions managing snapshots using the QEMU monitor.

QEMU monitor snapshot commands include:

| Command | Purpose |
|---------|---------|
| `savevm NAME` | Create a VM snapshot |
| `loadvm NAME` | Restore a VM snapshot |
| `delvm NAME` | Delete a VM snapshot |
| `info snapshots` | List VM snapshots |

Example:

```text
(qemu) savevm before-upgrade
(qemu) info snapshots
(qemu) loadvm before-upgrade
(qemu) delvm before-upgrade
```

Important limitations:

- VM snapshots require at least one non-removable, writable block device that supports snapshots, commonly a qcow2 disk image
- snapshots are not a replacement for backups
- restoring a VM snapshot reverts VM state and can affect running applications
- `-snapshot` mode stores temporary disk changes separately and discards them when QEMU exits unless changes are committed

> **Common Mistake**
>
> Do not confuse QEMU monitor snapshots with complete backup strategy.
> A snapshot is useful for rollback, but it may still depend on the original storage file.

# QEMU Guest Agent

The **QEMU Guest Agent** is a daemon that runs inside the guest operating system.

It allows the host or management layer to request guest-aware operations.

Typical uses include:

- retrieving guest information
- freezing and thawing filesystems
- improving snapshot consistency
- setting the guest time
- shutting down or suspending the guest
- reporting guest IP addresses to management tools

The guest agent is not part of the hypervisor.

It must be installed and running inside the guest.

---

## Guest Agent Communication

The QEMU Guest Agent usually communicates through a virtio-serial channel.

Conceptually:

```text
Host management tool
        |
QEMU character device
        |
virtio-serial channel
        |
qemu-ga inside guest
```

A QEMU command line may include a virtio serial device and channel for the agent.

Example:

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -device virtio-serial \
    -chardev socket,path=/tmp/qga.sock,server=on,wait=off,id=qga0 \
    -device virtserialport,chardev=qga0,name=org.qemu.guest_agent.0
```

Inside a Linux guest, the guest agent commonly uses:

```text
/dev/virtio-ports/org.qemu.guest_agent.0
```

The service name depends on the guest distribution.

Common Linux service names include:

```bash
systemctl status qemu-guest-agent
systemctl status qemu-ga
```

> **Exam Tip**
>
> The QEMU Guest Agent requires two sides:
>
> - a channel exposed by QEMU
> - the `qemu-ga` daemon running inside the guest

# VirtIO Device Drivers

**VirtIO** is a paravirtualized device framework.

Instead of emulating old physical hardware, QEMU can expose efficient virtual devices designed for virtual machines.

Common VirtIO devices include:

| Device | Purpose |
|--------|---------|
| `virtio-blk` | Paravirtualized block storage |
| `virtio-scsi` | Paravirtualized SCSI storage |
| `virtio-net` | Paravirtualized network adapter |
| `virtio-balloon` | Memory balloon device |
| `virtio-rng` | Random number generator device |
| `virtio-serial` | Communication channel, including guest agent transport |

Linux guests commonly include VirtIO drivers.

Other operating systems may require separate VirtIO drivers.

> **Exam Tip**
>
> VirtIO improves performance because the guest uses virtualization-aware drivers instead of relying only on fully emulated hardware devices.

# Installing and Maintaining QEMU

Package names vary by distribution.

Common package names include:

- `qemu`
- `qemu-system-x86`
- `qemu-kvm`
- `qemu-utils`
- `qemu-guest-agent`

Basic installation and maintenance tasks include:

1. Install QEMU system packages.
2. Ensure CPU virtualization is enabled in firmware.
3. Verify KVM modules are loaded.
4. Verify `/dev/kvm` exists and permissions are correct.
5. Start a test VM with `qemu-system-x86_64`.
6. Configure networking and storage.
7. Install the QEMU Guest Agent inside guests where needed.
8. Install VirtIO drivers inside guests where needed.

Useful checks:

```bash
qemu-system-x86_64 --version
qemu-system-x86_64 -machine help
qemu-system-x86_64 -accel help
lsmod | grep kvm
ls -l /dev/kvm
```

# QEMU Migration Concepts

The official objective description includes migrating QEMU installations.

At QEMU level, migration means transferring a running or paused VM from one QEMU process to another compatible QEMU process.

Migration usually requires:

- compatible QEMU versions
- compatible machine type
- compatible CPU model and CPU features
- compatible device configuration
- storage available on the destination or transferred separately
- network connectivity between source and destination

QEMU monitor migration commonly uses a destination started with `-incoming`.

Conceptual destination command:

```bash
qemu-system-x86_64 \
    -enable-kvm \
    -m 2048 \
    -drive file=vm.qcow2,format=qcow2,if=virtio \
    -incoming tcp:0.0.0.0:4444
```

Conceptual source monitor command:

```text
(qemu) migrate tcp:destination.example.com:4444
```

> **Boundary**
>
> Production migration is commonly managed by libvirt or a virtualization platform.
> Objective 351.4 covers libvirt migration in more detail.

# Troubleshooting QEMU

Troubleshooting QEMU means checking the correct layer.

Problems can occur in:

- CPU virtualization support
- KVM kernel modules
- `/dev/kvm` permissions
- QEMU command-line options
- disk image paths and formats
- storage permissions
- guest boot order
- guest drivers
- host networking
- guest networking
- QEMU monitor or guest agent communication

---

## QEMU Cannot Use KVM

Symptoms:

- QEMU reports that KVM is unavailable
- VM starts slowly because it uses TCG
- `-enable-kvm` fails

Check:

```bash
grep -E 'vmx|svm' /proc/cpuinfo
lsmod | grep kvm
ls -l /dev/kvm
qemu-system-x86_64 -accel help
```

Common causes:

- CPU virtualization disabled in firmware
- host CPU does not support required virtualization extensions
- KVM modules are not loaded
- current user cannot access `/dev/kvm`
- nested virtualization is not enabled when running inside another VM

---

## VM Does Not Boot

Symptoms:

- QEMU starts but no operating system boots
- guest firmware cannot find a boot device
- boot loops or kernel panics

Check:

- disk path
- disk format
- boot order
- installation ISO attachment
- guest storage driver support
- whether the guest expects BIOS or UEFI

Useful options:

```bash
-boot d
-cdrom installer.iso
-drive file=vm.qcow2,format=qcow2,if=virtio
```

---

## Guest Disk Is Missing

Symptoms:

- guest installer sees no disk
- installed guest cannot find root filesystem
- guest reports I/O errors

Common causes:

- wrong `-drive` path
- wrong image format
- missing permissions
- guest lacks VirtIO storage driver
- host storage is full

Check:

```bash
ls -lh vm.qcow2
df -h
ps aux | grep qemu
```

In the QEMU monitor:

```text
(qemu) info block
```

---

## Guest Network Does Not Work

Symptoms:

- guest has no network interface
- guest has no IP address
- guest cannot reach other systems
- port forwarding does not work

Check:

- `-netdev` option
- `-device` option
- matching `id=` and `netdev=`
- TAP device state
- bridge membership
- host firewall rules
- guest driver support

Useful commands:

```bash
ip link show
ip addr show
bridge link
brctl show
```

In the QEMU monitor:

```text
(qemu) info network
```

---

## Guest Agent Does Not Respond

Symptoms:

- management tool cannot query guest information
- filesystem freeze operations fail
- guest IP address is not reported

Check:

- whether the guest agent channel is configured in QEMU
- whether the VirtIO serial device exists
- whether `qemu-ga` is installed in the guest
- whether the guest agent service is running
- whether security policy blocks the agent

Inside a Linux guest:

```bash
systemctl status qemu-guest-agent
ls -l /dev/virtio-ports/
```

# Summary

QEMU is the userspace virtual machine process and device emulator used in the Linux virtualization stack.

When QEMU is combined with KVM, KVM provides hardware-assisted CPU virtualization through the Linux kernel while QEMU provides the VM process and virtual devices.

The important KVM modules are `kvm`, `kvm-intel` and `kvm-amd`, and QEMU accesses KVM through `/dev/kvm`.

The main command for x86_64 system virtualization is `qemu-system-x86_64`.

Important command-line options include `-enable-kvm`, `-m`, `-smp`, `-drive`, `-netdev`, `-device`, `-monitor`, `-cdrom`, `-boot`, `-nographic` and `-machine accel=kvm`.

QEMU networking commonly uses user-mode networking for simple setups or TAP devices connected to Linux bridges for more integrated networking.

QEMU storage maps guest block devices to disk images, block devices or other host storage backends.

The QEMU monitor can inspect and control a running VM and provides snapshot commands such as `savevm`, `loadvm`, `delvm` and `info snapshots`.

The QEMU Guest Agent runs inside the guest and enables guest-aware host operations.

VirtIO drivers provide efficient paravirtualized devices for storage, networking and other VM functions.

LPIC-3 also expects awareness of `ip`, `brctl` and `tunctl` in the context of QEMU networking.
